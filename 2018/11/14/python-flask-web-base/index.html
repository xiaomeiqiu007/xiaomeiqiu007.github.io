[Flask开发准备]：

Python虚拟环境介绍与安装：

因为python的版本更新的太快，flask的版本不兼容，一个虚拟环境对应一个falsk，就可以解决这个问题。

  1. 安装虚拟环境：pip install virtualevn

  2. 将虚拟环境安装到指定目录：virtualevn flask-evn

  3. 激活flask ：进入安装的目录进入Scripts 输入active

  4. 在虚拟环境中安装flask：pip install flask

Pycharm的使用：

创建一个新项目：选择刚才创建环境中的/bin/python.exe。也可以让pycharm自己创建一个flask虚拟环境

第一个flask应用：

flask中的session操作

flask中的session是存储在cookie中，然后用户拿着session来请求。来达到用户请求。

g属性

g:global。即全局属性，import g 添加某个变量的话g.username=“admin”。注意作用域是一次请求

钩子函数

1.before_request:在请求之前运行，在视图函数执行之间执行的，是一个装饰器。在每次请求之前执行，通常进行身份验证。

2.context_processr:为了解决渲染的时候多次传输统一变量。也是一个装饰器。只要返回一个字典，这个字典中的所有值，所有的模板都可以获取到返回的字典。

[flask_script]

1.Flask-Script:作用是可以通过命令行的形式来操作Flask。例如通过命令跑一个开发版本的服务器，设置数据可，定时任务等

2.如果在住mange.py写命令，那么在终端就只只需要``python manage.py` command_name就可以了

3.如果把一些命令集中在一个文件中，那么在终端就需要输入一个父命令，比如`python manage db init`

[Flask-Migrate]

1.因为采用db\_create\_all在后期修改字段的时候，不会自动映射到数据库中，必须删除表，然后重新运行\db_creatte_all`,才会重新映射，这样不符合我们的需求，flask-migrate就是为了解决这个问题，他可以在每次修改模型后，可以将修改的东西映射到数据可中。

2.安装pip install flask-migrate

代码实例：

# coding: utf-8
# auth:小煤球
from flask_script import Manager
from flask_migrate import Migrate,MigrateCommand
from untitled5 import app
from exts import db
from  models import User


# 模型 --》 迁移文件  --》表
# 建立模型：manage.py db init :初识化迁移脚本所需要的环境，只需要执行一次
# 生成迁移文件：manage.py db migrate :生成迁移脚本忙（会生成一个versions文件夹）
# 映射到表：manage.py db upgrate ：将迁移文件映射到表

#另外需要注意将数据模型导入

#  1.要使用flask_migrate,必须绑定qpp和db,绑定的db以及app必须是同一个
#  2. 把MigrateCommand命令添加到manager中
manager = Manager(app)
migrate = Migrate(app,db)
manager.add_command('db',MigrateCommand)

if __name__ == '__main__':
    manager.run()

[装饰器]：

闭包函数：当一个函数func1内部又定义一个函数func2，func2被称为内部函数，但是当内部函数的又存在用到func1中的临时变量，并且外函数的返回值是内函数的引用。这样就构成了一个闭包。

闭包函数示例：

# coding: utf-8
# auth:小煤球
def login(func):
    def regist():
        print 'tiis is regist'
        func()
    return regist
def run():
    print "this is run"
login(run)()

#首先会login函数将run作为参数，然后碰到内部函数，接着进行返回，返回的时候发现自己的临时变量func被内部函数用着那，没办法结束资源那只能将临时变量的值赋给内部函数自己先返回，login(run)就返回regist，发现regist()是个函数，那就执行，于是输出“this is regist”，接着输出run()函数中的内容。

因为这样写的太难看，装饰器的到来改变了这一写法。原来的程序改写为

# coding: utf-8
# auth:小煤球
def login(func):
    def regist():
        print 'tiis is regist'
        func()
    return regist

@login
def run():
    print "this is run"

run()

装饰器晋级（两个装饰器）：

def makeBold(fun):
    print('----a----')

    def inner():
        print('----1----')
        return '<b>' + fun() + '</b>'

    return inner


def makeItalic(fun):
    print('----b----')

    def inner():
        print('----2----')
        return '<i>' + fun() + '</i>'

    return inner


@makeBold
@makeItalic
def test():
    print('----c----')
    print('----3----')
    return 'hello python decorator'


ret = test()
print(ret)

输入结果：

----b----
----a----
----1----
----2----
----c----
----3----
<b><i>hello python decorator</i></b>

对结果进行分析：

  1. 装饰时机 通过上面装饰时机的介绍，我们可以知道，在执行到@makeBold的时候，需要对下面的函数进行装饰，此时解释器继续往下走，发现并不是一个函数名，而又是一个装饰器，这时候，@makeBold装饰器暂停执行，而接着执行接下来的装饰器@makeItalic，接着把test函数名传入装饰器函数，从而打印’b’，在makeItalic装饰完后，此时的test指向makeItalic的inner函数地址，这时候有返回来执行@makeBold，接着把新test传入makeBold装饰器函数中，因此打印了’a’。
  2. 在调用test函数的时候，根据上述分析，此时test指向makeBold.inner函数，因此会先打印‘1‘，接下来，在调用fun()的时候，其实是调用的makeItalic.inner()函数，所以打印‘2‘，在makeItalic.inner中，调用的fun其实才是我们最原声的test函数，所以打印原test函数中的‘c‘，‘3‘，所以在一层层调完之后，打印的结果为hello python decorator
